# 프로젝트 위그드라실

## 개요

프로젝트 위그드라실은 소설, 게임, 기타 창작물을 포함해 다양한 형태로 확장될 수 있는 창작 세계관 설정 프로젝트다.

## 프로젝트 구조

- `룩스테라`: TRPG 팀의 D&D 기반 캠페인에 사용될 배경으로 설계된 판타지 장르 하위 프로젝트다. 현재 가장 적극적으로 제작이 진행된 상태다.
- `엘드로스`: 포스트 아포칼립스 헌터물을 무대로 하는 하위 프로젝트다. 기반 설정이 이미 존재하며, 추후 본격적으로 확장할 예정이다.

## 세계관 관계

- `룩스테라`와 `엘드로스`는 서로 완전히 분리된 독립 세계가 아니라, 기본적으로 같은 세계관 계통 위에 놓인 연결된 프로젝트다.
- 다만 두 프로젝트는 동일한 시공이나 동일한 무대를 공유하는 것이 아니라, 서로 다른 시공과 무대를 가지는 프로젝트로 이해한다.
- 앞으로 추가되는 다른 하위 프로젝트들도 같은 세계관 계통 안에서, 서로 다른 시공 또는 무대를 가진 형태로 이어질 가능성이 높다.

## 현재 기준

- 현 시점에서는 `룩스테라`가 중심 개발 대상이다.
- `엘드로스`는 후속 확장 대상으로 간주하고, 기존 기반 설정을 보존한 채 나중에 내용을 추가한다.

## 시리즈 공통 폴더 구조

각 하위 프로젝트는 되도록 아래 폴더 구조를 공통 기준으로 따른다.

- `비밀 설정`: 아직 공개 전인 설정을 모아두는 폴더다.
- `설정`: 공개된 설정을 정리하는 폴더다.
- `신격`: 공개된 신격 정보를 정리하는 폴더다.
- `이미지`: 각 하위 프로젝트에서 사용하는 이미지 자료를 정리하는 폴더다.
- `템플릿`: 각종 설정 양식을 보관하는 폴더다.
- `설정 정비`: `설정` 및 `신격`을 기반으로, 신규 적용 사항이나 변경 예정 사항을 정리하는 폴더다.
- `스토리 설계`: 이야기 전개와 사건 설계를 다루는 폴더다.
- `세력 관계`: 세력, 인물, 단체 사이의 관계성을 정리하는 폴더다.

템플릿 문서는 `설정_기본 템플릿`, `설정_국가 템플릿`, `설정_단체 템플릿`, `설정_대륙 템플릿`, `신격_기본 템플릿`처럼 분류와 대상을 드러내는 방식으로 이름 짓는다.

현재 `룩스테라`의 `템플릿` 폴더에는 아래 템플릿이 존재한다.

- `설정_기본 템플릿`: 범용 설정 문서 작성 양식이다.
- `설정_국가 템플릿`: 국가 설정 작성 양식이다.
- `설정_단체 템플릿`: 단체 설정 작성 양식이다.
- `설정_대륙 템플릿`: 대륙 설정 작성 양식이다.
- `신격_기본 템플릿`: 범용 신격 문서 작성 양식이다.

## 룩스테라 문서 작성 지침

`룩스테라`에서 새 문서를 만들 때는 아래 메타 항목과 분류 체계를 기본 기준으로 삼는다. 아래 내용은 기존 문서 구조를 바탕으로 정리한 것이므로, 실제 문서 작성 시에는 문맥에 맞게 유도리 있게 적용한다.

### 공통 메타데이터 기준

원본 문서 메타데이터는 아래 공통 키를 기본 기준으로 사용한다.

- `uuid`: 모든 문서의 고유 식별자다. 제목, 파일명, 링크가 바뀌어도 유지한다.
- `title`: 문서 제목이다. 공통 제목 키는 `이름`이 아니라 `title`을 사용한다.
- `docType`: 앱과 후속 파이프라인에서 문서군을 판별하는 값이다.
- `thumbnail`: 목록 카드 대표 이미지다. 값이 없으면 반드시 빈 문자열 `''`로 둔다.
- `status`: 문서 진행 상태다.
- `created_at`: 자동 생성 항목으로 본다.
- `updated_at`: 자동 갱신 항목으로 본다.

현재 기본 `docType` 분류는 아래를 우선 기준으로 삼는다.

- `setting`: 설정 문서
- `deity`: 신격 문서
- `secret_setting`: 비밀 설정 문서
- `relation`: 세력 관계 문서
- `story_design`: 스토리 설계 문서
- `maintenance`: 설정 정비 문서
- `template`: 템플릿 문서

### 비밀 설정 문서

비밀 설정 문서는 문서 상단의 제목/메타 줄 대신 YAML 프론트매터를 기본으로 사용한다.

- `title`: 문서 제목이다.
- `secret_rank`: `하급`, `중급`, `상급`, `최상급`, `태고` 중에서 선택한다.
- `secret_type`: `인물`, `물건`, `장소`, `개념`, `사건` 중에서 선택한다.
- `status`: `시작 전`, `진행 중`, `완료` 중에서 선택한다.
- `created_at`: 자동 생성 항목으로 본다.
- `updated_at`: 자동 갱신 항목으로 본다.

### 설정 문서

설정 문서는 문서 상단의 제목/메타 줄 대신 YAML 프론트매터를 기본으로 사용한다.

- `title`: 문서 제목이다.
- `uuid`: 모든 설정 문서의 고유 식별자다.
- `docType`: `setting`을 사용한다.
- `thumbnail`: 대표 이미지가 없으면 빈 문자열 `''`를 사용한다.
- `status`: `시작 전`, `임시작성`, `진행 중`, `완료` 중에서 선택한다.
- `type`: `종족`, `사건`, `인물`, `아이템`, `지역`, `대륙`, `국가`, `도시국가`, `단체`, `주요 개념`, `개념`, `신격`, `도시`, `지형`, `부족국가` 중에서 선택한다.
- `subtype`: 필요할 경우 `항구`, `주요`, `일반`, `연맹`, `대륙`, `지침` 중에서 선택한다.
- `region`: 필요할 경우 `에리디안`, `발가르`, `청연`, `네베아`, `실란도르`, `칼디라`, `녹턴`, `카르코사`, `범대륙`, `진홍의 해협`, `심연의 의회` 중에서 선택한다.
- `created_at`: 자동 생성 항목으로 본다.
- `updated_at`: 자동 갱신 항목으로 본다.

### 설정 정비 문서

설정 정비 문서는 문서 상단의 제목/메타 줄 대신 YAML 프론트매터를 기본으로 사용한다.

- `title`: 문서 제목이다.
- `progress`: `준비중`, `작성중`, `작성완료` 중에서 선택한다.
- `written_on`: 날짜 또는 날짜 범위를 기록한다.
- `updated_at`: 자동 갱신 항목으로 본다.

### 신격 문서

신격 문서는 문서 상단의 제목/메타 줄 대신 YAML 프론트매터를 기본으로 사용한다. 본문은 `# 개요` 이하 내용을 유지하는 것을 원칙으로 한다.

- `title`: 문서 제목이다.
- `uuid`: 모든 신격 문서의 고유 식별자다.
- `docType`: `deity`를 사용한다.
- `thumbnail`: 대표 이미지가 없으면 빈 문자열 `''`를 사용한다.
- `pantheon`: 체크 여부로 표시한다.
- `cheongyeon_pantheon`: 체크 여부로 표시한다.
- `status`: `시작 전`, `진행 중`, `완료` 중에서 선택한다.
- `rank`: 현재 확인된 값 기준으로 `반신격`, `소신격`, `중신격`, `대신격`, `주신격`, `초신격` 등을 사용한다. 일부 중간 값은 아직 정리가 덜 된 상태이므로, 필요 시 기존 문맥을 우선 참고해 유도리 있게 적용한다.
- `alignment`: 현재 확인된 값 기준으로 `선신`, `회색신`, `악신` 등을 사용한다. 이 역시 일부 값은 추가 확인이 필요하므로 문맥에 맞게 적용한다.
- `domain`: 문자열 배열로 기록한다. 현재 확인된 값 기준으로 `빛`, `정화`, `어둠`, `원소`, `생명`, `죽음`, `질서`, `전쟁`, `수호`, `개척`, `지식`, `신비`, `운명`, `행운`, `부패`, `의지`, `평화`, `자연`, `재앙`, `시간`, `광기`, `여행`, `화로`, `예술`, `복수`, `꿈`, `대장간`, `황혼`, `폭풍`, `무덤`, `혼돈`, `기만` 등을 사용한다.
- `portfolio`: 문자열 배열로 기록한다. 현재는 고정 선택지 체계라기보다 사실상 자유 문자열 목록에 가깝게 운용한다. 일부 확인된 값으로는 `천명`, `천계 관료제`, `황권`, `예법`, `천문율`, `왕조의 승인`, `질서`, `역천`, `마도`, `강자존`, `약육강식` 등이 있다.
- `created_at`: 자동 생성 항목으로 본다.
- `updated_at`: 자동 갱신 항목으로 본다.

### 세력 관계 문서

세력 관계 문서는 문서 상단의 제목/메타 줄 대신 YAML 프론트매터를 기본으로 사용한다.

- `title`: 문서 제목이다.
- `relation`: `우호`, `견제`, `대립`, `전쟁` 중에서 선택한다.
- `a_setting`: 설정 문서를 가리키는 옵시디언 위키 링크로 기록한다.
- `a_god`: 신격 문서를 가리키는 옵시디언 위키 링크로 기록한다.
- `b_setting`: 설정 문서를 가리키는 옵시디언 위키 링크로 기록한다.
- `b_god`: 신격 문서를 가리키는 옵시디언 위키 링크로 기록한다.
- `issue`: 한 줄 요약 텍스트를 적는다.
- `session_signs`: 세션에서 드러날 단서나 징후를 텍스트로 적는다.
- `visibility`: `공개`, `부분 공개`, `비공개` 중에서 선택한다.

- `written_on`: 날짜 또는 날짜 범위를 기록한다.

### 표현 원칙

- 문서 지침에서는 실제 작성 대상 문서 기준 표현을 사용한다.
- 대신 `비밀 설정 문서`, `설정 문서`, `신격 문서`, `세력 관계 문서`, `스토리 설계 문서`처럼 실제 작성 대상 문서 기준으로 표현한다.
- 새 문서를 만들 때는 가능하면 문서 상단에 YAML 프론트매터를 두고, 본문은 그 아래에서 시작한다.
- `룩스테라\템플릿`의 템플릿 문서들도 같은 기준으로 YAML 프론트매터를 사용한다.
- `quartz/content`는 원본 문서의 심볼릭 링크 성격으로 보고, 메타데이터 정비의 직접 대상은 원본 문서만으로 본다.
- 문서 파일명 끝의 노션 32자리 아이디는 유지하지 않는다.
- 내부 링크도 아이디 없는 최종 문서명 기준으로 정리한다.
- 원본 노션 URL은 허용하지 않으며, 매핑이 애매한 경우 수동 검토 목록으로 분리한다.
- `엘드로스 임시` 같은 구버전 보관 폴더는 정리 대상에서 제외한다.
- `세력 관계`의 연결 항목은 노션 URL이 아니라 옵시디언 위키 링크인 `[[문서명]]` 형식으로 기록한다.

## 엘드로스 문서 작성 지침

`엘드로스`에서 새 문서를 만들 때는 기본적으로 `룩스테라`의 문서 작성 기준을 공통 골격으로 사용한다. 다만 장르와 설정 구조 차이로 인해, 아래 분류 기준은 `엘드로스` 쪽 문맥에 맞춰 별도로 적용한다.

현재 `엘드로스`의 `템플릿` 폴더에는 아래 템플릿이 존재한다.

- `설정_종족 템플릿`: 종족 설정 작성 양식이다.
- `설정_몬스터 템플릿`: 몬스터 설정 작성 양식이다.

- `신격_기본 템플릿`: 범용 신격 문서 작성 양식이다.

- `status`: `시작 전`, `임시작성`, `진행 중`, `완료` 중에서 선택한다.
- `type`: `종족`, `사건`, `인물`, `아이템`, `지역`, `대륙`, `국가`, `도시국가`, `단체`, `주요 개념`, `개념`, `신격`, `도시`, `지형`, `부족국가`, `클래스` 중에서 선택한다.
- `subtype`: 필요할 경우 문서 성격에 맞는 세부 분류를 사용한다. `엘드로스`에서는 `시스템`, `길드`, `클랜`, `장비`, `기기`, `역사` 같은 값이 우선 후보가 된다.
- `region`: 필요할 경우 작품 내 실제 권역 체계가 정리된 뒤 그 기준에 맞춰 사용한다.
- `created_at`: 자동 생성 항목으로 본다.
- `updated_at`: 자동 갱신 항목으로 본다.

### 분류 원칙

- `클래스` 개별 문서는 `type: 클래스`를 사용한다.
- `클래스` 개별 문서는 파일명을 `클래스-문서명` 형식으로 두고, `type: 클래스`를 사용한다.
- `클래스` 체계 자체를 설명하는 총괄 문서는 `type: 주요 개념`을 사용한다.
- `헌터`, `게이트`, 각성 구조처럼 세계 규칙이나 작동 원리를 설명하는 문서는 우선 `type: 개념` 또는 `type: 주요 개념`을 기준으로 판단한다.
- `길드`, `클랜`처럼 단체 성격이 분명한 문서는 `type: 단체`를 유지하고 필요 시 `subtype`으로 세분화한다.
- `길드` 개별 문서는 `type: 단체`, `subtype: 길드`를 우선 기준으로 삼는다.
- `클랜` 개별 문서는 `type: 단체`, `subtype: 클랜`을 우선 기준으로 삼는다.
- 기술물이나 장비성 사물은 모두 우선 `type: 아이템`으로 정리하고, 세부 구분이 필요하면 `subtype`으로 보완한다.
- `몬스터` 문서는 별도 최상위 타입으로 분리하지 않고 `type: 종족`, `subtype: 몬스터`를 기본 기준으로 사용한다.
- `도시` 개별 문서는 `type: 도시`를 사용한다.
- `인물` 개별 문서는 `type: 인물`을 사용한다.
- `사건` 개별 문서는 `type: 사건`을 사용한다.

## 구조 적용 기준

- `룩스테라`는 이미 이 구조를 따르고 있으므로, 폴더명은 공통 기준에 맞게 정리해 사용한다.
- `엘드로스`는 아직 본격적인 내용이 비어 있어도, 같은 구조의 폴더 골격은 미리 구성해 둔다.

## graphify

This project has a knowledge graph at graphify-out/ with god nodes, community structure, and cross-file relationships.

When the user types `/graphify`, invoke the `skill` tool with `skill: "graphify"` before doing anything else.

Rules:
- For codebase questions, first run `graphify query "<question>"` when graphify-out/graph.json exists. Use `graphify path "<A>" "<B>"` for relationships and `graphify explain "<concept>"` for focused concepts. These return a scoped subgraph, usually much smaller than GRAPH_REPORT.md or raw grep output.
- Dirty graphify-out/ files are expected after hooks or incremental updates; dirty graph files are not a reason to skip graphify. Only skip graphify if the task is about stale or incorrect graph output, or the user explicitly says not to use it.
- If graphify-out/wiki/index.md exists, use it for broad navigation instead of raw source browsing.
- Read graphify-out/GRAPH_REPORT.md only for broad architecture review or when query/path/explain do not surface enough context.
- After modifying code, run `graphify update .` to keep the graph current (AST-only, no API cost).
